### 第一阶段：素材收集与提取

以下按维度对搜索材料进行要点提取：

#### 1. 按类型、按功能、按层、按领域、按模块组织目录的资料

- **按类型组织**（按功能类型分包）：将代码按技术角色分目录——`components/`、`pages/`、`utils/`、`apis/`、`store/`。这种结构的优点是简单直观，适合小型项目或快速原型开发，但在复杂大型应用中会导致功能逻辑分散、业务边界模糊、修改时需跨多目录搜索，且难以扩展。[reference:0]

- **按层组织**（分层架构 / Layered Architecture）：将代码按架构层划分——`presentation/`（表现层）、`business_logic/`（业务逻辑层）、`data_access/`（数据访问层），或 MVC 模式中的 `models/`、`views/`、`controllers/`。按层分包是组织代码的常见方式，有助于职责分离，提升代码可维护性。[reference:1][reference:2]

- **按功能组织**（垂直代码组织 / Vertical Code Organization）：一种将代码按照业务功能而非技术层次进行划分的方法。所有与特定功能相关的代码集中在一起，包括前端组件、后端逻辑、数据库模型等，形成内聚的功能模块。Wasp-lang 项目的实践验证了其多项优势：更高的内聚性、更简单的维护（功能模块可作为完整单元进行修改或删除）、更清晰的架构（功能边界明确）、更好的可测试性。[reference:3][reference:4]

- **按领域组织**（领域驱动设计 DDD / Feature-first）：一级目录为业务模块（feature），每个 feature 自包含所有相关逻辑——组件、页面、状态管理、服务、类型定义。如 PawHaven 项目中 `features/RescueDetail` 和 `features/ReportStray` 各自独立管理。[reference:5][reference:6]

- **按模块组织**（模块化 / Modular Architecture）：将项目拆分为独立模块（packages），每个模块自治。这在 monorepo 场景中体现为 `apps/`、`packages/` 的顶层划分，每个模块边界清晰、可独立开发和部署。[reference:7]

#### 2. 文件树视觉清晰度、根目录拥挤、目录边界、模块自治的观点

- **根目录拥挤问题**：不应该把 curl 之类的目录放到根目录下，这样会给人误导，使得接手的人某天增加新功能时也会新增一个文件夹到根目录，随着功能越来越多，根目录会逐步混乱。对系统的代码和目录预先设计好、规划好目录结构，什么目录放哪种类型的文件，这是可维护性的基础。[reference:8]

- **目录边界**：每个模块通过一个 `index.ts` barrel file 与外界通信，外部只能通过这个唯一入口访问模块内部。dependency-cruiser 可以静态分析 import 图，在 CI 中自动阻止违反边界规则的导入。[reference:9]

- **模块自治**：Feature 目录下的每个业务模块高度自包含，团队成员可独立开发、测试和部署。模块间通过明确的公共 API 交互，内部实现细节对外不可见。[reference:10][reference:11]

- **文件树视觉表达**：GitHub Next 的可视化实验指出，代码库的可视化应当提供足够的"指纹"（fingerprint），让观看者扫一眼就能看到代码库的结构。树图（tree maps）、节点链接图、圆形打包等可视化方式均可用于表达嵌套的目录层次。[reference:12]

#### 3. 可近似反映模块组织的可见代理指标

- **顶层目录数量**：从根目录看，顶层目录的数量直接反映项目的组织粒度。太少（如 0-2 个）意味着扁平或未组织，太多（如 20+ 个）意味着根目录拥挤。

- **根目录文件散布**：根目录中散落的文件数量（如 `.config`、`main.py`、`README.md` 等不应大量散布的文件）是一个直观的代理指标，反映项目是否经过基本的目录规划。[reference:13]

- **功能目录聚合**：功能相关的文件是否聚集在同一目录下，而非分散在 `components/`、`utils/`、`services/` 等多处，直接决定了视觉上的"功能边界"是否可见。RFC 0007 中提出重组 client components 从 flat 结构到 feature-based 结构，正是为了解决"相关代码散落在不同目录中"的问题。[reference:14]

- **层级深度与同质性**：Structure101 使用 containment hierarchy（包含层级）来度量物理结构，从高层文件夹向下逐层组织。一个衡量标准是目录内部文件的同质性——即同一目录下的文件是否属于同一逻辑组件。[reference:15]

#### 4. 反例：小项目、单文件工具、框架强约定目录、monorepo 场景

- **小项目 / 单文件工具**：早期 Go 项目常以单一 `main.go` 起步，根目录干净，但这并不代表"最佳实践"，只是规模决定的自然状态。[reference:16]

- **框架强约定目录**：MVC 框架如 Rails、Django、Laravel 等会强制按类型组织目录（`controllers/`、`models/`、`views/`），这种结构是框架约定而非开发者主动选择，且可能被量表误判为"较低档位"。文章指出"比如 MVC 的框架，定好了 controllers，views，models，后面刚入门的人，都能按照目录结构去加代码"，这说明了约定式目录的价值。[reference:17]

- **Monorepo 场景**：Monorepo ≠ monolith。一个 monorepo 的顶层可能是 `apps/`、`packages/` 等业务无关的技术目录，但其内部每个 package 都可能是高内聚低耦合的模块。按外表判断 monorepo 的"视觉清晰度"可能产生误判——顶层看起来像"按类型组织"，但内部每个 package 都是自治的。Monorepo 的关键特征是一致共享历史和一个连贯的依赖图，而非"许多文件夹"。[reference:18][reference:19]

- **Monorepo 不应被简单线性评判**：Monorepo 本身不是"更好"或"更高级"的选择，而是需要在收益和增加的复杂性之间仔细权衡后采用。[reference:20]

#### 5. 关键启发总结

- **可观察性优先**：该维度衡量的应是"在 IDE 左侧文件树中肉眼可见的目录组织方式"，而非项目实际采用了哪种架构模式。文件树的视觉特征包括顶层目录数量与类型、文件散布程度、目录命名模式、边界清晰度。

- **策略不等于线性阶梯**：按类型、按层、按功能、按领域、按模块是不同的组织策略（organizing strategies），而非同一维度上的线性进阶。将它们串成 0→5 档的阶梯存在概念混淆。

- **外部约定不应影响评分**：框架强约定的目录结构（如 Rails 的 `app/models/`）和 monorepo 的技术顶层目录（如 `packages/`）是外部约束，不应被线性量表简单判断为"好"或"坏"。

- **根目录是"第一印象"** ：根目录的整洁程度（顶层目录数量、散落文件）是文件树视觉清晰度最直观的代理指标。

- **边界可见性**：功能/领域/模块之间的边界是否在文件树中"一目了然"，是区分不同组织成熟度的关键视觉特征。

### 第二阶段：现有文案逐档审查

#### 名称审查：「模块与文件夹组织 (Module / Folder Organization)」

名称准确捕捉了核心概念——文件夹组织方式与模块划分方式，但"模块"一词在 0-5 档中的含义飘移不定：0 档无模块，1 档按类型，2 档按功能，3 档按层，4 档按领域，5 档按模块。建议将名称微调为 **"文件树与目录组织 (File Tree & Directory Organization)"** ，更直接指向可观察的视觉对象。

#### 0-5 档文案分析

| 档位 | 当前文案 | 组织策略 | 问题诊断 |
|------|---------|---------|---------|
| **0** | 左侧文件树像杂乱的地摊，各种扩展名的文件散落在根目录，毫无分组，找文件只能靠记忆和搜索 | 无组织 | ✅ 准确，视觉特征清晰——根目录散布、无分组 |
| **1** | 文件树出现按类型命名的文件夹，根目录变清爽，但各组内部仍可能混杂 | 按类型 (Type-based) | ✅ 合理，但"按类型"是否必然优于"无组织"？对于极小型项目，0 可能反而合理 |
| **2** | 项目按特征切成功能村落，每个村落内相关文件紧密聚集，视觉上形成明显功能分界 | 按功能 (Feature-based) | ⚠️ "功能村落"的隐喻是否一定比"按层"更高级？实际上按层（MVC）在很多场景中被视为经典实践 |
| **3** | 分层结构清晰可见，屏幕左侧依次排列 models/views/controllers 等层叠目录 | 按层 (Layer-based) | ⚠️ **核心问题**：将按层（MVC）排在按功能之后（第 2 档之后）意味着"按功能优于按层"，这在软件工程中没有公认共识 |
| **4** | 领域驱动聚合成型，目录以业务领域命名，每个领域内部自成体系、边界分明 | 按领域 (Domain/DDD) | ⚠️ 按功能（第 2 档）与按领域（第 4 档）本质上是同一策略的不同成熟度表达，而非两种独立策略。DDD 只是"按功能"的更强纪律版本 |
| **5** | 高度模块化的微型城市，顶层出现独立模块目录，各模块完全自治，视觉上呈现相互隔离又工整排列的模块群 | 按模块 (Module/Package) | ⚠️ 模块化≠领域驱动。模块化关注的是包级自治与技术边界，领域驱动关注的是业务语义边界。两者是不同的维度 |

#### 核心问题总结

1. **策略混淆为线性阶梯**：将按类型→按功能→按层→按领域→按模块写成一条线性进阶路径，但它们是不同的组织策略。按层（MVC）与按功能（垂直切片）是平行的设计选择，并非层级关系。搜索材料指出垂直代码组织与传统水平分层（如 MVC 模式）是两种不同的方法，各有利弊，没有绝对的优劣之分。[reference:21]

2. **2 档与 4 档概念重叠**：按功能（"功能村落"）与按领域（"DDD"）是同一个维度上不同严格程度的表达——DDD 就是按功能组织但加上更严格的边界纪律，不应拆成两档。

3. **3 档与 2/4 档的顺序争议**：按层（MVC）在软件工程历史上是经典的、被广泛验证的组织方式。将其放在按功能之后（暗示它是更高级的形态）缺乏公认依据。实际上很多成熟项目是在按层的基础上叠加按功能的。

4. **5 档引入不同维度**：模块自治强调依赖方向和接口隔离，关注的是模块间的耦合关系，它与文件树的外观有重叠但并非同一维度。一个按层组织的项目也可以有良好的模块自治（通过严格的 import 规则），一个按领域组织的项目也可能模块间耦合严重。

5. **视觉可观察性不足**：部分档位的描述偏离了"代码长什么样"的核心——"各模块完全自治"不是肉眼在文件树中能看到的，需要通过静态分析工具（dependency-cruiser）检查 import 图才能确认。[reference:22]

### 第三阶段：评估单调性、清晰度、互斥性

- **单调递增** ⚠️ 部分成立（0→1→2→3 可视递增，但 2→3 的策略顺序有争议，3→4→5 递进逻辑断裂）。
- **清晰度** ⚠️ 3-4 档的"按层"与"按领域"之间的区分在视觉上不够清晰。按层的文件树中 `models/` 内部也可能按领域进一步细分。
- **互斥性** ❌ 不成立。按功能与按领域高度重叠；按模块与按领域有交叉；按层可与按功能叠加。各档位描述的不是互斥的状态，而是不同策略的组合。

### 第四阶段：修改方案

重新设计量表的核心思路：**统一为"从左到右文件树中可见的目录组织方式，从无组织到有组织到强自治"的单一可观察维度**，放弃将不同策略线性排列的做法。

#### 替代文案（仅限本维度 6 条）

| 档位 | 文案 |
|------|------|
| **0** | 左侧文件树根目录杂乱，文件散落无目录分组，找文件只能靠文件名记忆或全局搜索 |
| **1** | 根目录出现少数顶层文件夹（如 `src/`、`docs/`），但内部仍以扁平文件为主，尚未形成二级目录层次 |
| **2** | 顶层目录按技术类型清晰分区（如 `components/`、`utils/`、`services/`），文件树呈现整齐的类型聚簇，根目录整洁 |
| **3** | 目录内部出现嵌套层级（如 `src/` 下按功能或按层进一步划分），文件树呈现稳定的二级或三级深度，视觉上出现"目录套目录"的层次感 |
| **4** | 各功能/业务域在文件树中有明确的可视边界（如 `features/order/` 与 `features/user/` 完全分离），每个域内部文件紧密聚合，目录名即业务语义 |
| **5** | 顶层模块目录高度自治（如 monorepo 的 `packages/` 下各模块），每个模块有独立的 `src/`、`test/`、`package.json`，视觉上呈现完全对等且边界封闭的模块群 |

### 第五阶段：修改说明

1. **统一为"视觉可观察"维度**：所有描述均聚焦于 IDE 左侧文件树中肉眼可见的结构特征——目录是否存在、顶层命名模式、嵌套深度、边界是否清晰、模块是否对等封闭。

2. **消除策略线性化的概念混淆**：不再将"按类型""按层""按功能""按领域"写成线性阶梯。改为描述目录组织从扁平到层次、从无结构到有结构、从模糊到边界清晰的渐变过程。

3. **2→3→4 形成逻辑递进**：2 档（按类型分区）→ 3 档（出现嵌套层级）→ 4 档（业务域边界可视化）→ 5 档（模块自治且边界封闭）。每一步都是在前一步基础上的结构深化，而非策略跳跃。

4. **保留根目录整洁度作为门槛**：0→1→2 描述根目录从混乱到整洁的渐变，这一线索在搜索结果中被反复强调为可维护性的第一道门槛。[reference:23]

5. **DDD 不再作为独立档位**：DDD 作为一种设计方法被融入"业务域边界可视化"的描述中（第 4 档），而非作为独立的线性阶梯。

6. **Monorepo 作为第 5 档的典型场景**：在第 5 档中明确提及 monorepo 的 `packages/` 模式，这是"模块自治"在文件树中的最强视觉表达。
