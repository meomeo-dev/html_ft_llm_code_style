## 维度审查报告：代码复用强制力 (Code Reusability Enforcement)

### 总体评估

**该维度名称与量表描述之间的一致性存在中度偏差。** 量表0-4级较好地遵循了从代码外观（代码长什么样）直接观察的原则，但第5级引入了静态检查工具（static analysis checker）注解和绝对化DRY教条，偏离了“肉眼可见的代码外观”这一核心观察维度。以下逐项展开审查。

---

### 1. 维度名称是否准确表达“代码复用强制力”的代码外观？

**维度名称「代码复用强制力」(Code Reusability Enforcement) 与量表实际测量内容存在张力。** “Enforcement”一词暗示某种**强制执行机制（enforcement mechanism）**——可能是工具、流程或治理规则在发挥作用，而非单纯的代码外观。公开资料中，“enforcement”通常与静态分析规则（static analysis rules）、CI检查策略（check-in policies）或代码审查门禁相关联[reference:0]。

然而，该量表的0-4级实际描述的是**代码复用程度在视觉上的递进表现**：从复制粘贴块（copy-pasted blocks）→ 提取函数（extracted method）→ 公共模块导入（common/util import）→ 内部库目录（internal lib directory）。这些属于**代码复用行为的可观察外观（observable code appearance of reuse）** ，而非“强制力”本身。

**结论**：若量表严格限定在代码外观维度，建议名称调整为更贴近“可见复用程度”或“复用可见性”的表述；若保留“强制力”，则必须厘清是指“复用行为在代码中的可视化痕迹”，而非工具或流程层面的强制执行。

---

### 2. 0-5档是否在同一条视觉维度上单调递进？

**第0-4级：在视觉维度上单调递进，递进逻辑清晰。**

| 等级 | 视觉外观 | 关键视觉信号词 |
|------|---------|--------------|
| 0 | **大段完全相同代码块**（identical code blocks），复制粘贴痕迹（copy-paste traces）明显 | 无函数提取（no extract method） |
| 1 | **已有两次重复代码块**（duplicated blocks twice）仍未提取 | 相同逻辑在多处出现（same logic in multiple locations） |
| 2 | **重复逻辑被提取为独立函数**（extracted into standalone function），原位置替换为函数调用（function call） | 抽取后单行调用（single-line invocation） |
| 3 | **所有重复代码迁移至公共模块**（migrated to common module），业务文件简洁 | import 来自 common、util 等包 |
| 4 | **项目构建了内部库/工具集**（internal library/toolkit），统一lib目录 | 复用以引入库的形式出现 |

这一递进链与已知的代码复用演化路径一致：从识别重复代码（duplicated code）→ 通过提取方法重构（Extract Method refactoring）消除局部重复[reference:1]→ 将共享功能集中在公共库（common libraries）中实现跨项目复用[reference:2]。

**第5级：出现维度跳跃。** “任何重复都不可见，静态检查工具注解提示DRY严格执行，每一段逻辑只出现一次”引入了三个问题：
- **静态检查工具注解（static checker annotation）** 是工具配置层面的存在，不一定在代码文件中直接可见（除非注解以注释形式嵌入源码）。
- **“任何重复都不可见”** 和 **“每一段逻辑只出现一次”** 属于绝对化表述（DRY教条），不符合现实工程实践——重复代码检测工具如CPD（Copy-Paste-Detector）通常允许设定阈值，因为并非所有相似代码都需要消除[reference:3][reference:4]。
- 该级描述超出了**代码外观**范畴，进入了**流程治理（process governance）** 和**工具强制执行（tool enforcement）** 领域。

---

### 3. 跨维度内容识别

**以下内容被识别为偏离“代码外观”维度：**

#### 3.1 静态检查强制（Static Analysis Enforcement）
原量表第5级的“静态检查工具注解提示DRY严格执行”将关注点从“屏幕上看到什么”转移到“工具配置了哪些规则”。静态分析工具（如PMD/CPD、jscpd、CodeScene）虽然能检测重复代码（duplicate code detection），但它们的配置文件和运行结果不属于代码文件本身的视觉外观[reference:5][reference:6]。该内容应属于**质量保障流程（quality assurance process）** 或**工具治理（tool governance）** 维度，而非代码外观量表。

#### 3.2 “任何重复都不可见”的绝对化表述
DRY原则（Don‘t Repeat Yourself）在实践中被误解为“零重复”，但原始定义是“每一份知识在系统内拥有单一、明确、权威的表示”（Every piece of knowledge must have a single, unambiguous, authoritative representation within a system）[reference:7]。代码外观无法判断“每一段逻辑只出现一次”——这需要语义分析和全局知识建模，属于**设计质量（design quality）** 维度的判断，而非屏幕可见信息。

#### 3.3 第4级到第5级是否属于“代码复用强制力”的合理延伸？
若将“强制力”理解为**复用行为的可见终点**，则第5级应描述“复用完全落地后屏幕上的代码特征”，例如：
- 代码库中无法通过肉眼定位到大段重复逻辑；
- IDE中重复检测插件的标记已清零；
- 所有业务文件中的逻辑均通过调用公共模块实现。

**静态检查注解属于“规则如何被检查”，而非“代码长什么样”。**

---

### 4. 替代文案建议（6条，全部基于代码外观直接观察）

以下替代方案严格遵循 **“屏幕上能看到什么”** 的观察原则，剔除了静态检查强制、流程治理、DRY绝对化等跨维度内容：

---

**等级0 — 大段复制粘贴 (Rampant Copy-Paste)**

> **视觉信号词：大段重复代码块（large duplicated code blocks）、无函数提取（no extract method）。** 屏幕上出现大段完全相同的代码块，复制粘贴痕迹（copy-paste traces）明显，逻辑以连续重复的语句块形式存在，未见任何抽取后的函数调用（extracted function call）替代。

---

**等级1 — 重复出现但未处理 (Duplication Accumulated Without Extraction)**

> **视觉信号词：重复代码块（duplicated code blocks）、相同逻辑多处出现（same logic in multiple places）。** 屏幕上至少有两处包含相同逻辑的代码块，但未提取为独立函数。肉眼可以定位到相似片段（similar-looking fragments）在不同文件或不同位置反复出现，形成可辨识的重复模式（repetition pattern）。

---

**等级2 — 局部提取函数调用 (Local Extract Method Applied)**

> **视觉信号词：提取函数（extracted method）、单行函数调用（single-line function call）。** 重复逻辑已被提取为独立函数，原重复代码位置被替换为清晰的单行函数调用。屏幕上可见 `extractMethodName()` 形式的调用替代了原本的大段重复块，函数定义（function definition）出现在同一文件或同模块内。

---

**等级3 — 公共模块集中复用 (Shared Common/Util Module Import)**

> **视觉信号词：common/util 导入（common/util import）、跨文件函数调用（cross-file function invocation）。** 业务文件中的复用逻辑不再以本地函数形式存在，而是通过导入公共模块实现。屏幕上可见大量 `import { ... } from '@/common/...'` 或 `from '@/util/...'` 等导入语句，业务代码本身简洁，共享逻辑集中在统一的公共模块（common module）目录下。

---

**等级4 — 内部库/工具集目录可见 (Internal Library/Package Directory Visible)**

> **视觉信号词：lib目录（lib directory）、内部工具集（internal toolkit）、包引入（package import）。** 项目文件树中可见独立的 `lib/`、`packages/` 或类似目录结构，其中包含可复用的内部库或工具集（internal library/toolkit）。业务文件通过引入库的形式复用逻辑，屏幕上可见 `import { ... } from '@myproject/lib/...'` 或等效的包路径导入语句。

---

**等级5 — 重复逻辑肉眼不可见 (No Duplicate Logic Visually Detectable)**

> **视觉信号词：无肉眼可辨识重复块（no visually detectable duplicate blocks）、代码库中逻辑唯一性可见（visibly unique logic）。** 在代码审查的屏幕阅读范围内，无法通过肉眼定位到大段相似逻辑。每个业务功能的实现路径唯一，文件间逻辑边界清晰，偶有相似片段可通过上下文判定为独立语义（independent semantics）而非复制粘贴产物。IDE内重复检测插件（如CPD/JetBrains Duplicates）的标记区域为空或仅有琐碎匹配，但**该等级不要求也不描述任何静态分析工具的配置或注解存在与否**——仅描述代码文件在屏幕上的最终视觉状态。

### 总结

原量表的核心问题在于第5级将“代码复用”的视觉外观升级偷换为“静态检查强制执行”的流程治理概念，同时引入了DRY教条的绝对化表述。建议将量表严格收敛到 **“代码在屏幕上的可观察复用形态”** 这一单一维度，确保每个等级都能通过肉眼或IDE内标准视图直接判定，无需查阅工具配置文件、CI流水线设置或团队流程文档。
