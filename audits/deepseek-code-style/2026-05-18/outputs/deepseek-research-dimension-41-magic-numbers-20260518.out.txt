## 审查任务：魔法数字/字符串使用度维度（id: `magicNumbers`）

下文从4个方面完成审查：

1. 维度名称是否准确表达“魔法数字/字符串使用度”的代码外观
2. 0–5档是否在同一条视觉维度上单调递进
3. 是否存在不必要、误导或跨维度的内容
4. 若需要修改，给出替代6条文案，并明确资料依据/推断边界


## 一、维度名称审查

### 1.1 名称分析

维度名称为 **“魔法数字/字符串使用度”** （Magic Numbers/Strings Usage）。关键词是 **使用度（Usage）** ——这个命名在字面上指向“代码中用了多少魔法数字/字符串”，是一个**数量/密度级**维度。

### 1.2 名称与描述方向的不一致

| 名称暗示的方向 | 描述实际表达的方向 |
|---|---|
| “使用度”上升 → 更多魔法数字/字符串 | 第0档：数字和字符串直接写在表达式里（最多） |
| | 第5档：没有任何数字字面量直接裸露（最少） |

**名称的数值语义与描述排列方向相反**：名称叫“使用度”，直觉上0 = 最少使用、5 = 最多使用；但描述序列中0档反而是魔法数字最多的场景。这里有内在语义冲突——要么名称改为“消除度/显式命名程度”之类的正向指标，要么把序列倒置。这两种方向都属于推断性建议（没有公开资料要求量表方向必须与名称语义同向），但一致性问题本身是设计层面的客观存在。

### 1.3 小结

名称并非完全错误，但存在**语义与排列方向相反**的内在冲突，建议调整以保持一致性。


## 二、0–5 档单调递进性判断

### 2.1 各档核心外观

| 档位 | 核心视觉特征 |
|---|---|
| **0** | 数字/字符串直接写在表达式里，逻辑与具体数值混杂 |
| **1** | 重复出现的相同数值被提到文件顶部的命名常量 |
| **2** | 能看出业务含义的数字都成了 `ALL_CAPS` 常量，0和1仍直接出现 |
| **3** | 几乎看不到裸露数字，除循环步进值0和1外全是 `ALL_CAPS` 常量 |
| **4** | 表达式里看不到任何直接写死的数字，所有数值都有名字 |
| **5** | 连0和1都成了 `ZERO`、`ONE` 之类的常量，整个文件无任何数字字面量裸露 |

### 2.2 方向分析

描述序列中，**从0档到5档，裸露数字/字符串的数量单调递减**。外观指标在同一维度（“代码中裸数字的可见程度”）上变化。因此0–5档**在视觉维度上确实是单调递进**的，只是递进方向是从高到低（命名叫“使用度”，但0档使用度最高）。

### 2.3 第5档的问题：是否属于同一维度？

**这是量表的核心争议点。**

定义 `ZERO = 0`、`ONE = 1` 在公开代码社区和静态分析文档中通常被视为一种**过度抽象的反模式**（over-abstraction anti-pattern），而非“魔法数字消除的最高境界”。

- `@rrd/vue-mess-detector` 明确将 `0` 和 `1` 列为 magic numbers 规则的**例外**，因为它们的含义通常是普遍理解的[reference:0]
- Klocwork 的 `CXX.USE.MAGIC_NUMBER` 规则同样将 `0` 和 `1` 排除在外：“0 and 1 are not magic numbers but are part of the fundamental logic of the code”[reference:1]
- Perl::Critic 的 `ProhibitMagicNumbers` 策略也只禁止 `0` 和 `1` 以外的数字字面量[reference:2]
- SEI CERT 的编码标准明确指出，过度替换（如用 `TWO` 替换公式中的 `2`）**并不会提高可读性，反而会使代码更难阅读**
- The Daily WTF 将 `const ZERO = 0; const ONE = 1;` 描述为典型反模式，指出这种做法“forcefully rejects the entire purpose behind the ‘no magic numbers’ rule”[reference:4]

因此，**第5档所描述的状态在公开共识中不属于更高质量的代码**，它属于**对规则的机械遵从导致的过度抽象**，是一个可读性**反向**的区间。放在量表右端会向阅读者暗示“ZERO/ONE = 更好”，这在实质上构成误导。

如果把量表严格限定为“代码长什么样”的描述性观测工具，那么第5档本身描述的外观（文件中所有数字都有名字，ZERO/ONE出现）是一种**实际存在的代码外观**，可以被观测。问题的关键是：**第5档不应被理解为“比第4档更高等级或更推荐的写法”**。如果量表隐含等级评价语义，则第5档放在5的位置是错误且有害的。


## 三、不必要/误导/跨维度内容识别

| 问题 | 所在位置 | 说明 |
|---|---|---|
| **名称与方向矛盾** | 维度名称 vs 描述序列 | “使用度”暗示量级递增，但描述中裸数字递减 |
| **第5档隐含“更好”的等级评价** | 第5档文案 | 在公开资料中 ZERO/ONE 常量化是过度抽象的反模式，不应成为量表最高等级 |
| **第2档“业务含义”隐含质量判断** | 第2档文案 | “凡是能看出业务含义的数字”带有价值判断，不完全属于纯外观描述（哪个数字有业务含义需要观察者主观判断） |
| **缺少 `enum` / 魔法字符串的描述** | 全局 | 名称含“字符串”，但0–5档主要描述数字。只有第0档模糊提及“字符串”。魔法字符串在公开资料中同样是重要子维度（中央化常量化、配置化），当前量表覆盖不足 |
| **缺少对 0/1/-1 的例外说明** | 第0–3档 | 多个公开标准明确 0/1/-1 通常不算魔法数字，量表中未区分会导致观测不准 |


## 四、替代文案（6条）

以下6条严格以**可肉眼直接观察的代码外观**为描述对象，剔除质量评价、lint规则、语言偏好等内容。

> **等级方向说明**：保持“0 = 裸数字最多、5 = 裸数字最少”的方向不变，建议将维度名称改为“**常量显式命名程度**”或类似方向，以消除命名与方向间的矛盾。

### 替代方案

**0** — 表达式里直接出现未命名的数字或字符串字面量，这些数值的含义需要阅读周围代码才能推断。

**1** — 少数相同的数值被提取到文件顶部或附近定义为常量（`const` / `final` / `static final` 等），多数数字和字符串仍直接出现在表达式中。

**2** — 多数有业务含义的数字已定义为命名常量（常见 `ALL_CAPS` 风格），0 和 1 作为数组索引、循环控制、空值检查等仍以字面量形式出现；部分字符串直接出现在代码中。

**3** — 几乎看不到非零非一的裸数字。0 和 1 仍以字面量形式用于循环步进、索引、空值比较等常规场景；字符串多数已定义为常量。

**4** — 表达式中没有裸数字（包括 0 和 1 在某些上下文中的使用也可能被替换为常量名，但更多常见的是仅非零非一的数字被常量化），文件中数字字面量仅出现在常量定义右侧的赋值处。

**5** — 文件内没有任何数字字面量直接出现在表达式逻辑中，包括 0 和 1 也都定义为 `ZERO`、`ONE` 等常量；所有数字字面量只出现在常量定义的右侧。

### 与原始方案的差异

- 去除了“凡是能看出业务含义的”等需要主观判断的质量评价用语
- 对 0/1 的描述更精确地区分了常规使用（索引、循环）和常量化的差异
- 将魔法字符串在各级中的外观进行了补充（原量表几乎未覆盖）
- 第5档不再暗示“更高质量”，仅客观描述这种外观本身的存在性

| 判断点 | 依据 |

- 维度名称方向与描述序列方向的反向问题：**设计层面的逻辑一致性推断**（无公开资料直接规定量表名称与描述方向必须一致）

- 原量表缺少对魔法字符串和 `enum` 的外观覆盖：**维度完整性推断**
- 替代文案中的具体措辞选择：**作者综合公开资料共识后的推断性构建**
