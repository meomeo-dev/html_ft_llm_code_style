**回复ID: parameterCount-20260518-001**

## 一、维度名称审查

**维度名称（Dimension Name）**：`函数/方法参数数量限制 (Parameter Count Limit)`

该名称准确表达了“从小括号内可直接数出的形参个数”这一**代码外观（Code Appearance）**。行业共识中，"Parameter Count Limit" 是 linter 规则的标准命名——ESLint 的 `max-params` 规则默认限制函数定义中最大参数数量为 3[reference:0]，Checkstyle 的 `ParameterNumber` 检查默认最多 7 个参数[reference:1]，SwiftLint 的 `function_parameter_count` 规则默认限制为 5[reference:2]。名称本身是**中性且可观察的（Observable）**，不涉及 API 设计质量判断，无需修改。

---

## 二、0-5 档单调递进性判断

量表 0-5 档在**同一条视觉维度上呈单调递进（Monotonically Progressive）**，总体方向是从“参数多/乱”到“参数少/规整”：

| 档位 | 外观特征概括 | 递进方向 |
|------|-------------|---------|
| 0 | 参数随意排列，长长一串挤在一起，偶有空无一物 | 最混乱 |
| 1 | 参数个数通常看得过去，偶尔较长但大多不超一行 | ↑ |
| 2 | 很少超过 4 个，超过时往往换行缩进对齐 | ↑ |
| 3 | 参数列表短小，超过 3 个被改成大括号包裹的对象 | ↑ |
| 4 | 几乎每个函数 0-2 个参数，单个参数居多 | ↑ |
| 5 | 要么无参，要么只有一个参数，且该参数几乎总是数据类或对象字面量 | 最精简 |

递进逻辑清晰：**参数数量递减（Parameter Count Decreasing）→ 换行对齐格式化（Line-wrapping Alignment）→ 参数对象引入（Parameter Object Introduction）→ 趋向单参或无参**。每一步都是从肉眼可见的代码外观上可区分的。

## 三、跨维度内容识别与风险分析

### 3.1 第 5 档的核心问题

第 5 档描述为：

> “函数要么无参，要么只有一个参数，且该参数几乎总是一个数据类或对象字面量”

这里的 **“数据类或对象字面量（Data Class / Object Literal）”** 存在跨维度风险（Cross-dimension Risk），需要精细辨析：

**它可以是参数数量限制维度的自然延伸**：当团队严格执行“参数数量 <= 1”时，原本需要多个参数的数据必须被打包进一个对象。此时，对象字面量是**强制参数上限后的必然外观结果**，不是 API 风格选择。代码肉眼可见：`function doSomething(params)` 而非 `function doSomething(a, b, c, d)`。

**但它也可能滑入 API 风格/参数对象模式维度**：Options Object 模式是一种有意识的设计选择，目的是让参数命名化、可选化、顺序无关[reference:3][reference:4]。它不属于参数数量限制维度——团队可能允许 5 个参数，但出于可读性偏好仍使用 options object。

**判定标准**：
- 如果代码库中几乎所有多参数函数都必然被打包成对象，且这是 lint 规则强制的结果（如 `max-params = 1`），则属于本维度。
- 如果对象字面量的使用是风格偏好（即便参数只有 2 个也用对象包装），则属于 API 风格/参数对象模式维度。

建议在第 5 档中明确强调 **“这是参数上限强制后的必然外观”** ，而非“偏好使用对象字面量”。

### 3.2 第 3 档的类似问题

第 3 档：

> “参数列表都很短小，超过 3 个的情况被改成大括号包裹的对象”

“被改成大括号包裹的对象”同样涉及 **参数对象重构（Introduce Parameter Object）** 这一重构手法。Martin Fowler 在《重构》中将“长参数列表”识别为代码坏味道，建议使用“引入参数对象”重构[reference:5]。该重构的执行**原因**可能是逻辑考虑（高内聚），但外观**结果**是：小括号内参数少、大括号包裹对象。只要量表聚焦外观结果而非重构动机，就不算越界。

### 3.3 低档位（0-1）的模糊性

第 0 档：

> “函数小括号里参数随意排列，有的长长一串挤在一起，有的空无一物”

“有的空无一物”与“长长一串挤在一起”并存，表明**代码库内参数数量不一致**——这是编码规范缺失的典型外观信号。这与高档位的“一致地少”形成递进，逻辑自洽。

第 1 档：

> “参数个数通常看得过去，偶尔有函数包含较长的参数列表，但大多不超过一行”

“大多不超过一行”引入了**行长度（Line Length）** 维度（代码行最大字符数）。但在这里，它是参数数量在视觉上的自然代理（Proxy）：参数少→不换行→一行内。这种使用是可接受的，因为没有单独测量行长度，而是将其作为参数数量视觉外观的辅助描述。

### 3.4 总结：不越界的项目

以下内容**没有**出现在量表中，设计正确：
- ❌ 未涉及 API 设计质量（如“API 是否易用”）
- ❌ 未涉及 lint 工具强制（如“ESLint 报错”）
- ❌ 未涉及语言特定风格（如“Go 语言选项模式”）
- ❌ 未涉及设计质量（如“函数内聚性”）

## 四、替代文案建议（6 条，全部基于代码外观）

以下 6 条替代文案均为**直接从代码外观观察（Directly Observable from Code Appearance）** 的描述，避免了对 API 质量、参数对象偏好、lint 强制、设计质量或语言风格的判断。

> **第 0 档**：**函数签名（Function Signature）** 小括号内参数随意排列，数量波动剧烈，有的长长一串挤在一行，有的空无一物，无统一约定。  
> **第 1 档**：参数个数偶有超标，个别**长参数列表（Long Parameter List）** 出现，但大多数函数签名仍可保持在一行内，整体勉强看得过去。  
> **第 2 档**：**参数数量（Parameter Count）** 很少超过 4 个；一旦超过，调用者会采用**换行缩进对齐（Line-wrapping and Indentation Alignment）**，签名跨多行但排列工整。  
> **第 3 档**：小括号内参数数量整体短小；原本需要多参数的场景，肉眼可见地被改写为**大括号包裹的参数对象（Parameter Object）**，即单个对象字面量替代了多个独立参数。  
> **第 4 档**：几乎每个**函数定义（Function Definition）** 都只有 0-2 个参数，单个参数的情况占大多数，双参偶尔出现，三参及以上极少见到。  
> **第 5 档**：函数要么无参，要么只有一个参数——且这是参数数量上限强制执行后的必然外观，该唯一参数几乎总是一个数据类或对象字面量。

| 原档位 | 原描述核心问题 | 修改策略 |
|--------|---------------|---------|
| 0 | “空无一物”与“长长一串”并存，可能暗示不一致 | 明确“无统一约定”，强化“数量波动剧烈” |
| 1 | “大多不超过一行”可能引入行长度维度 | 保留，作为参数数量少的自然代理 |
| 2 | 无问题 | 仅做措辞优化 |
| 3 | “被改成”暗示动机 | 改为“肉眼可见地被改写为”，聚焦外观结果 |
| 4 | 无问题 | 仅做措辞优化 |
| 5 | “数据类或对象字面量”可能误读为风格偏好 | 明确“这是参数上限强制后的必然外观” |
