## 一、维度名称准确性审查

### **维度名称准确表达了"输入验证严格度"的代码外观**

**Input Validation Strictness（输入验证严格度）** 这个名称聚焦于代码中"能否看到验证逻辑"的可见程度，完全从代码外观维度出发。OWASP 代码审查指南指出输入验证是"最有效的应用安全技术控制之一"，代码审查时应关注"所有外部输入都被检查和验证"[reference:0]。该名称准确概括了从"看不到任何验证"到"存在显式验证管线"的视觉递进。

---

## 二、0-5 档单调递进性审查

### **六档在同一视觉维度上单调递进，但第1档存在定位偏差**

| 档位 | 描述摘要 | 视觉特征 | 评价 |
|------|----------|----------|------|
| **0** | 参数直接传入业务逻辑，无任何检查 | 代码路径上完全没有验证语句 | 基准，正确 |
| **1** | 仅前端可见 `required`/`maxlength`/`pattern` | HTML 属性或 JS 校验，后端入口无验证 | **存在问题**（见下文） |
| **2** | 后端出现 `if`/`typeof`/`length`/正则等内联检查 | 控制器或路由中出现逐字段判断块 | 正确 |
| **3** | `ALLOWED_VALUES`/`enum`/`Set.has()`/`includes()` 白名单对比 | 输入点旁可见白名单数据结构 | 正确 |
| **4** | 独立 schema/validator 定义，入口处调用 `parse()`/`validate()` | 声明式 schema 定义 + 显式调用 | 正确 |
| **5** | 请求链中多个命名验证中间件或 `assert`/`check` 形成管线 | 可识别的中间件链或断言调用链 | 正确 |

### **第1档的问题：前端校验不应占据独立档位**

第1档（前端校验）和第2档（后端内联检查）之间并非严格的**同一维度上的单调递进**。前端校验和后端校验属于**不同的执行位置**而非验证严格度的递进——OWASP 明确指出"客户端验证可被轻易绕过，服务端验证才是安全基础"[reference:1]。一个只有前端校验的系统严格度应被视为**等同于第0档**（无有效验证），而非独立档位。

**建议**：将第1档修改为"后端入口处存在零散的、不完整的类型或存在性检查（如单个 `if (x)` 判空）"，使递进关系完全落在后端验证的严格度轴上。

## 三、跨维度内容识别

### **以下内容存在跨维度风险**

**1. 前端校验的独立档位（第1档）问题**

如前所述，前端校验是**部署位置**概念，不是**严格度**概念。当前量表将"前端有后端无"设为第1档，容易导致审阅者将"优秀的前端校验体验"误判为"有一定的输入验证严格度"——这与量表目标冲突。

**2. "白名单"（Allowlisting）与"黑名单"（Denylisting）的语义陷阱**

第3档使用"白名单对比"是正确的代码外观描述——审阅者可以看到 `ALLOWED_VALUES`、`Set.has()` 等数据结构。但需警惕审阅者将"黑名单过滤"（如 `str.replace(/<script>/g, '')`）也纳入此档——黑名单属于**输出净化/过滤**而非输入验证。OWASP 明确区分 allowlisting 与 denylisting，指出 allowlisting 是推荐的最小方法[reference:2]。

**3. 业务规则未混入当前量表——但需保持警惕**

当前 0-5 档均未涉及业务规则。但若未来修订，需注意 OWASP 区分"数据验证"（Data Validation）与"业务验证"（Business Validation）——业务验证关注"值范围或交易是否合理"，属于业务逻辑层[reference:3]。业务规则混合输入验证是一种代码异味（Code Smell），应在量表说明中明确排除。

**4. 净化（Sanitization）未混入量表——正确**

输出净化（如 HTML 实体编码、SQL 参数化）与输入验证是不同概念——验证回答"数据是否符合预期格式"，净化回答"如何安全处理数据"[reference:4]。当前量表完全聚焦输入验证外观，未涉及净化，这是正确的。

## 四、替代文案建议

基于以上审查，推荐以下 6 条从代码外观直接观察的文案（修改了第 1 档，其余微调）：

> **0** — 请求参数或表单数据直接传入业务逻辑，代码路径上完全看不到任何 `if`、正则、断言、类型检查或验证库调用。
>
> **1** — 后端入口处存在零散的、不完整的存在性或类型检查（如单个 `if (x)` 判空或 `typeof` 检测），但未覆盖全部输入字段。
>
> **2** — 后端路由或控制器中出现系统性的 `if`/`typeof`/`length`/正则表达式等内联边界检查块，逐字段覆盖主要输入点。
>
> **3** — 输入点旁可见 `ALLOWED_VALUES` 常量、`enum` 定义、`Set.has()` 或 `includes()` 等白名单数据结构，用于精确约束合法值集合。
>
> **4** — 独立 schema 或 validator 定义成块（如 Joi / Zod / Pydantic 类型注解），入口处显式调用 `parse()` / `validate()` 等方法进行声明式校验。
>
> **5** — 请求处理链中可见多个命名验证中间件或 `assert`/`check` 调用，形成可识别的显式验证管线（Pipeline）。

### 修改说明

| 修改点 | 原描述 | 新描述 | 依据 |
|--------|--------|--------|------|
| 第1档重写 | 客户端校验 | 后端零散的存在性检查 | OWASP：前端校验不可依赖[reference:5]；客户端校验与严格度不构成递进关系 |
| 第2档强化 | "后端路由或控制器中出现" | 加入"逐字段覆盖主要输入点" | 区分零散检查（第1档）与系统性的内联检查（第2档） |
| 第3档微调 | "ALLOWED_VALUES、enum、Set.has() 或 includes()" | 加入"用于精确约束合法值集合" | 强调白名单的语义特征 |
| 第4档微调 | "独立 schema 或 validator 定义成块" | 加入"(如 Joi / Zod / Pydantic 类型注解)" | 提供具体可观察的代码模式 |
| 第5档微调 | "形成显式验证管线" | 加入"Pipeline" | 更精确描述中间件链的视觉特征 |

## 七、总结

当前量表的 **0、2、3、4、5 档** 构成了一条清晰的"输入验证严格度"视觉递进链，维度名称准确。核心问题在于 **第 1 档**（前端校验）打破了同一维度的递进逻辑——它测量的是验证代码的部署位置而非严格度。建议将第 1 档替换为"后端零散的存在性/类型检查"，使全部六档严格沿着"后端验证手段的强度和覆盖面"递增。替代文案已在上方给出，可直接替换使用。
