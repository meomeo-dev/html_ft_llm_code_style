# 输出清理 (Output Sanitization) 量表维度审查报告

## 执行摘要

**审查焦点**：本次审查聚焦于量表中 **输出清理（输出编码）代码外观维度** 的定义与递进逻辑，不涉及真实 XSS 防护有效性判断，也不涉及对特定模板引擎或框架的偏好。

**审查方法**：基于 OWASP 输出编码规则、上下文感知编码框架、以及模板引擎自动转义等公开资料，对原量表 0-5 档描述逐一审阅。

**核心发现**：
1. **维度名称** ＂输出清理 (Output Sanitization)＂ 在安全工程语境中容易与输入端的清洗/过滤混淆，建议修改。
2. **第 4 档（CSP 响应头）与第 5 档（统一中间件）** 属于 **跨维度偏移**（CSP 属于 HTTP 响应安全策略层，统一中间件属于架构治理层），二者均无法在单文件代码中直接通过肉眼观察判断，破坏了量表的递进纯粹性与可观测性。
3. 第 0-3 档构成了一条从 **裸拼接 → 基础转义 → 上下文分场景编码 → 模板隐式自动转义** 的清晰可见的视觉递进链，应予保留。

**建议**：将第 4 档（CSP）与第 5 档（统一中间件）从本维度移除，替换为更贴合代码外观视觉递进逻辑的替代描述。

---

## 一、维度名称审查

### 1.1 当前名称：＂输出清理 (Output Sanitization)＂

**存在的问题**：

在安全工程领域，**sanitization（清理）** 一词通常指对输入数据进行清洗、移除或修改危险内容的操作，例如 HTML 标签剥离、危险属性过滤等。**输出阶段** 的正确操作是 **encoding（编码）** 或 **escaping（转义）**，即将特殊字符转换为安全的等价表示形式（如 `&lt;` 代替 `<`），而非删除或修改数据内容。OWASP 明确区分了这两类操作：**Output Encoding** 用于在输出时将数据转换为安全的表示形式，而 **Sanitization** 更多指向对用户提交富文本进行清洗的独立操作[reference:0][reference:1]。

**推断**：＂Output Sanitization＂ 作为维度名称，可能导致使用者将 **编码/转义** 与 **输入清洗** 两种本质上不同的操作混淆。这属于术语使用的精确性判断，基于对 OWASP 知识体系的常识性理解。

**建议修改为**：＂输出编码 (Output Encoding)＂ 或 ＂输出转义 (Output Escaping)＂。在中文语境中，＂输出编码＂ 更为通用且与 OWASP 规范保持一致。

## 二、逐档审查与递进性分析

### 2.1 第 0-3 档：正确的视觉递进链 ✅

| 档位 | 原描述 | 可观察代码特征 | 判断 |
|------|--------|---------------|------|
| **0** | 屏幕上是字符串拼接，没有任何转义函数，视觉上用户数据直接嵌入 HTML | 裸 `+` 拼接、`${}` 插值、无任何 `escape`/`encode` 函数包裹 | ✅ 有效 |
| **1** | 出现 htmlEscape() 等基本转义调用，视觉上输出语句会包裹一层简短函数 | `htmlEscape(userName)`、`escapeHtml(comment)` 等单一函数包裹 | ✅ 有效 |
| **2** | 根据输出上下文分别可见 escapeHtml()、escapeJs()、encodeURIComponent() 等编码函数 | 同一文件出现多种编码函数，按使用位置区分 | ✅ 有效 |
| **3** | 模板文件视觉上使用自动转义语法，转义由引擎隐式完成，不再显式调用函数 | `{{ var }}`、`{% autoescape %}` 等模板标记，无显式 encode 调用 | ✅ 有效 |

**分析**：

这四档构成了一条 **清晰、单向递进的视觉演化链**：

**信号词：裸拼接（String Concatenation）→ 基础转义（Basic Escaping）→ 上下文编码（Contextual Encoding）→ 自动转义（Auto-Escaping）**

这条链的变化是肉眼可观察的：
- 第 0 档：肉眼可见的 `+`、`${}`、`innerHTML = userInput`，没有任何编码函数包裹[reference:2]
- 第 1 档：肉眼可见的 `escapeHtml(data)` 或 `htmlspecialchars(input)` 包裹在变量周围
- 第 2 档：同一文件/模板中肉眼可见 **多种编码函数**，分别对应 HTML 体、属性、JavaScript、URL 等不同输出位置——这正是 OWASP 所强调的 **上下文感知输出编码（Contextual Output Encoding）** 的代码外观[reference:3][reference:4]
- 第 3 档：模板文件视觉上 **不再出现显式编码函数调用**，而是通过模板引擎的语法标记（如 Handlebars 的 `{{ }}`、Jinja2 的 `{{ var }}`、Twig 的 autoescape 标签）隐式完成转义[reference:5]

**递进逻辑**：从 **无防护 → 有防护但简单 → 有防护且精细 → 有防护且自动化**，每一步的代码外观变化都可以在代码审查时直接观察。

### 2.2 第 4 档：Content-Security-Policy 的跨维度偏移 ❌

**原描述**：
> 输出代码旁边可见 Content-Security-Policy 头设置语句，配合转义函数共同出现

**问题诊断**：

**信号词：内容安全策略（Content-Security-Policy, CSP）属于 HTTP 响应头安全策略层，而非代码层输出编码**

CSP 在代码中的表现形式通常为：

```http
Content-Security-Policy: default-src 'self'; script-src 'none'
```

或服务端代码中的：

```go
w.Header().Set("Content-Security-Policy", "default-src 'self'")
```

这种代码外观与输出编码语句有本质区别：
1. **CSP 是独立的安全层次**：它是浏览器端的策略执行机制，属于 **纵深防御（Defense-in-Depth）** 的最后一层，而非输出编码本身[reference:6]。
2. **＂配合转义函数共同出现＂是偶然而非必然**：在代码审查中，CSP 头设置语句与具体输出编码语句之间没有语法上的必然共现关系。一个文件可能只设置 CSP 头而没有输出编码，反之亦然。
3. **跨维度污染**：CSP 属于 **HTTP 安全响应策略** 维度，与 **输出编码代码外观** 维度正交。将 CSP 纳入输出编码量表，会误导使用者认为 CSP 是输出编码的一种形式，这是危险的概念混淆。OWASP 明确指出 CSP 不应用于替代输出编码[reference:7]。

**结论**：本描述应被移除。建议替换为与输出编码代码外观直接相关的观察项。

### 2.3 第 5 档：统一中间件的跨维度偏移 ❌

**原描述**：
> 所有输出都通过统一的安全函数/中间件，屏幕上不会有任何直接的拼接或裸变量输出

**问题诊断**：

**信号词：中间件（Middleware）治理属于架构层，其代码外观跨文件分布，无法在单文件代码审查中直接判定**

诊断点：

1. **不可观察性**：＂所有输出都通过统一安全函数/中间件＂——这一判断需要对 **整个代码库进行全局分析**，审查所有输出路径是否都经过同一个函数。在单文件代码审查中，审查者只能看到当前文件是否调用了某个安全函数，但无法判断 **所有文件是否都一致使用了该函数**。这是架构治理层面的属性，而非单文件代码外观属性。

2. **中间件的代码外观特征**：中间件（如 Express.js 的 `app.use()`、Spring 的 `Filter` 或 `Interceptor`）的注册代码通常出现在应用启动/配置文件中，而非在业务代码输出语句旁边。这意味着中间件注册语句与具体输出语句在代码中 **物理上分离**，不符合本量表要求的＂屏幕上的代码外观＂。

3. **维度偏移**：统一中间件/统一输出函数属于 **架构一致性（Architectural Consistency）** 或 **安全基础设施（Security Infrastructure）** 维度，而非 **输出编码代码外观** 维度。它与＂字符串拼接→转义函数→上下文编码→模板自动转义＂这条视觉链处于不同的抽象层次。

**结论**：本描述应被移除。建议替换为聚焦于 **代码中可观察的输出编码高级模式** 的描述。

## 三、递进性总结

**第 0-3 档递进有效** ✅：从裸拼接到自动转义的视觉演化是单变量单调变化的——**输出语句中编码函数的可见性逐步降低**（从无函数→有函数→多种函数→无显式函数但由模板引擎隐式处理）。

**第 4-5 档递进断裂** ❌：CSP 和中间件不属于同一视觉维度。它们引入了新的变量（HTTP 头策略、架构层治理），破坏了单向递进性。

## 四、替代建议

### 4.1 建议的 6 条替代文案

以下替代方案均聚焦于 **可在单文件代码审查中通过肉眼直接观察的输出编码代码外观**：

| 档位 | 替代文案 | 可观察特征 |
|------|---------|-----------|
| **0** | 屏幕上是字符串拼接或模板插值，用户数据直接嵌入 HTML，无任何转义或编码函数包裹 | 裸 `+` / `${}` / `` ` `` 模板字面量 / `innerHTML = userInput` |
| **1** | 出现基础转义函数调用（如 htmlEscape / escapeHtml / htmlspecialchars），输出语句包裹一层简短函数 | `htmlEscape(name)` / `escapeHtml(comment)` |
| **2** | 根据输出上下文分别使用不同编码函数，可见 escapeHtml / escapeJs / encodeURIComponent 等并存在同一文件中 | `escapeHtml(title)`、`escapeJs(userJson)`、`encodeURIComponent(redirectUrl)` 共存 |
| **3** | 模板文件中使用自动转义语法（如 `{{ var }}`、`{% autoescape %}`），用户数据由模板引擎隐式转义，不再显式调用编码函数 | `{{ user.name }}` / `{% autoescape true %}` / `th:text="${user.name}"` |
| **4** | **（替代）** 输出前使用上下文感知编码器（如 OWASP Java Encoder 的 `Encode.forHtml()`、`Encode.forJavaScript()` 等），函数名明确指示目标上下文，调用点在输出位置就地可见 | `Encode.forHtmlAttribute(userInput)` / `Encode.forJavaScript(data)` 调用点与 `document.write` 或模板输出在同一作用域 |
| **5** | **（替代）** 输出变量被安全 HTML 包装器类型包裹（如 SafeHtml 对象），屏幕上变量类型明确标注为已编码的安全数据，而非原始字符串；输出语句直接引用安全包装器实例，无任何裸字符串变量 | 类型标注为 `SafeHtml` / `TrustedHTML`，变量名含 `safeHtml` / `escaped` 前缀，`element.innerHTML = safeHtml.toString()` |

### 4.2 各档替代理由

**第 4 档替代说明**：

**信号词：上下文感知编码器（Context-Aware Encoder）** 的代码外观与第 2 档的基础编码函数有明确的视觉递进区别。第 2 档是开发者自行选择不同函数，第 4 档是使用标准化编码器库（如 OWASP Java Encoder 的 `Encode.forHtml()` 系列），其 API 设计本身即强制了上下文感知，函数命名模式（`forHtml` / `forJavaScript` / `forCss`）在代码中清晰可见[reference:8]。

**此处判断基于公开资料**：OWASP Java Encoder 项目的 API 设计文档明确展示了上下文感知编码的函数命名规范。

**第 5 档替代说明**：

**信号词：安全包装器类型（Safe Wrapper Type）** 代表了输出编码的最高级代码外观——不再是通过函数调用来编码，而是通过类型系统在编译期保证输出安全。在代码审查中，审查者可以看到变量被声明为 `SafeHtml` 或 `TrustedHTML` 类型，而非普通 `string`。这与第 3 档（模板引擎隐式转义）的区别在于：第 3 档的隐式性体现在模板标记上，第 5 档的隐式性体现在类型系统上，后者在非模板场景（如纯 JavaScript/TypeScript 的 DOM 操作）中同样适用。

**此处判断基于推断**：SafeHtml 包装器模式在 Google Closure Templates、TypeScript 的 Trusted Types 等场景中有应用，但具体的类型命名和代码外观因框架而异。建议使用者根据实际技术栈调整类型名称。

### 4.3 关于原第 4 档 CSP 和第 5 档中间件的处理建议

- **CSP 响应头** 应归入独立的＂安全响应头（Security Response Headers）＂维度，与 HSTS、X-Frame-Options、X-Content-Type-Options 等组成单独的量表。
- **统一中间件/安全函数** 应归入＂安全架构一致性（Security Architecture Consistency）＂维度，与输入验证管道、统一异常处理等组成单独的量表。

上述归属建议均为推断，基于安全工程中对防御层次的一般性分类理解。

## 六、总结

**当前量表的核心问题**：第 4 档（CSP）和第 5 档（统一中间件）引入了与＂输出编码代码外观＂不直接相关的安全机制，导致维度定义膨胀、递进逻辑断裂。

**建议行动**：
1. 维度名称从＂Output Sanitization＂修改为＂Output Encoding＂；
2. 保留第 0-3 档；
3. 将第 4 档替换为＂上下文感知编码器调用＂的代码外观；
4. 将第 5 档替换为＂安全包装器类型＂的代码外观；
5. 将 CSP 和中间件相关描述移入各自专属的独立维度。
