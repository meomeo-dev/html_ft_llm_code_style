## 1. 概述：**单一职责原则（Single Responsibility Principle）的外观观察模式**

经由多轮检索与交叉比对，从主流设计原则资料、代码审查实践、静态分析指标以及过度拆分反例四个方向提取了与“屏幕可见外观”相关的信号。核心发现概括为以下关键词结构：**肉眼可辨识度（Naked-eye Identifiability）**、**块边界信号（Block Boundary Signal）**、**命名体量比（Name-to-body Ratio）**、**空行语义密度（Whitespace Semantic Density）**、**异质块可见性（Heterogeneous Block Visibility）**。

以下按维度逐一呈现搜索结果与评估，不涉及架构优劣、业务正确性或工程成熟度。

---

## 2. 各搜索方向的核心发现

### 2.1 SRP 在代码层的外观线索

微软官方文档将大函数定义为“膨胀物”（bloater），明确列出可肉眼观察的指标：超过 30 行即产生异味，80 行几乎总是过长；深层嵌套（循环内套循环、嵌套 if/else/try）；超过 5–6 个参数；高抽象层与低抽象层交织。此外，注释如 `// Step 1`、`// Step 2` 暗示多段异质逻辑，每个分段都可以是一个独立函数。JetBrains 的 Upsource 博客进一步指出：从代码审查侧-by-side diff 中可直观看出方法内混杂“消息解析”与“UI 渲染”两类性质完全不同的代码块[reference:2]。

### 2.2 混杂职责在外观上的表现

长方法（Long Method）典型的视觉表现是：方法过于冗长，包含多个逻辑分支和多个职责[reference:3]。方法名本身也是外观信号——“方法名里带‘and’或‘or’”意味着响应两个变化原因[reference:4]。**空行分段**被确认为核心视觉手段：空行本质上是“视觉分组”，但过度使用（每条语句间都塞空行）会打断阅读节奏，Clean ABAP 指南明确指出这种冲动“可能暗示你的方法没有做到只做一件事”[reference:5]。当屏幕上的函数被空行切割后，仍能看到个别块做了性质完全不同的事，这正是职责混杂的外观特征。

### 2.3 静态分析中可近似反映职责混杂的可见代理指标

传统静态分析指标中，**函数长度（Function Length）** 是最直接的视觉代理：长函数倾向于做太多事[reference:6]。**圈复杂度（Cyclomatic Complexity）** 虽然量化了逻辑分支数，但视觉上对应的表现是“凹凸不平的道路”（Bumpy Road）——函数内出现多个嵌套逻辑块，迫使阅读者频繁减速[reference:7]。**认知复杂度（Cognitive Complexity）** 更贴近视觉体验：嵌套越深，人眼越难以在一个屏幕内建立上下文。此外，“提取想象”测试——如果你能给一块代码命名一个函数名，它大概就应该是一个独立函数——本质上是个视觉块识别测试。

### 2.4 过度拆分的视觉噪声（反例）

过度原子化同样产生可见噪声。小到 1–3 行、仅被调用一次的函数被称为 **micro-methods**，读者需要在多个函数间跳转才能理解完整逻辑，“容易迷路、容易丧失优化视野”。当函数体远短于其名称和声明的开销时（如 `int foo() { return do_foo(); }`），视觉上产生大量“函数壳”碎片，反而掩盖逻辑流。Martin Fowler 的观点为边界提供了锚点：**名称与本体之间的语义距离**才是核心标准，而非单纯的函数长度[reference:10]。这意味着 5 档“高度对称、布满短小规律的函数列”在某些场景下可能是过度拆分的噪声，而非理想状态。

## 3. 当前文案的多维度评估

### 3.1 名称准确性：**单一职责原则的外观量表** vs. 用户意图

**评估结果：名称准确**。当前维度文案始终围绕“屏幕上可见的代码块边界”和“异质代码块混杂程度”展开，没有滑入“类的变化原因”“需求变更频率”等纯设计层面概念。但需要指出：维度名称“单一职责原则”本身是设计原则，量表化后应强调其**外观代理维度**（例如“职责边界可视化”或“代码块同质性”），但这不在本轮修文范围内，仅作提示。

### 3.2 单调性：0→5 是否单向递增

**评估结果：基本单调，但存在局部跳跃风险**。

- **0→1→2→3** 形成清晰递进：从“无任何分块”到“空行勉强分隔”，再到“函数明显变短变多、大多数意图清晰”，再到“每个函数/类专注如激光”。视觉上逐级“清爽”，可辨识。
- **3→4** 出现跳跃：4 档强调“严格原子化、寥寥数行、俳句集合”，引入了**函数体积极短小**这一新维度，部分偏向过度拆分的极端。
- **4→5** 引入了**模块内聚**和**函数列的规律性**，这些属于“组织结构视图”而非纯粹的“职责边界视图”，与 0–3 的核心线索（异质块混杂→清晰意图块）存在维度漂移。

### 3.3 概念混入风险

**评估结果：存在轻度混入，但尚未严重混淆**。

以下概念的可见外观与“职责边界”高度相关，但各档位侧重不同：

| 概念 | 出现档位 | 属于纯外观？ | 是否存在混入？ |
|---|---|---|---|
| 函数长度 | 0,1,2,3,4 | ✓ 是 | 合理，未混入 |
| 空行切割 | 1 | ✓ 是 | 合理 |
| 命名精准度 | 3 | 部分（命名可见，但“精准”是语义） | 轻微混入 |
| 过度原子化 | 4 | ✓ 是 | 合理 |
| 模块内聚 | 5 | **否**（模块内部关联强度，非纯视觉） | 混入风险 |
| 函数列规律性 | 5 | 部分 | 混入风险 |

### 3.4 是否存在架构/哲学/业务层面误导

**评估结果：未发现明显误导**。文案始终避免使用“变化原因”“行为者”“测试性”“可维护性”等架构术语，集中在“屏幕上的函数长什么样”。唯一可商榷的是 5 档中“模块内聚如乐高积木群”引入了模块级概念，弱化了函数级的外观观察。

### 3.5 档位间概念边界

**评估结果：0–3 边界清晰，4 与 5 存在部分重叠**。

- **0↔1**：清晰——有无空行切割的差别。
- **1↔2**：清晰——“仍有个别臃肿”vs“大多数只对应一个清晰意图”。
- **2↔3**：清晰——“个别臃肿”到“无臃肿”。
- **3↔4**：基本清晰——从“小但非极短”到“极短”，但语义空间较小。
- **4↔5**：**边界模糊**——4 的“寥寥数行、俳句集合”与 5 的“布满短小规律的函数列”高度重叠，区别在于 5 加入了模块内聚维度。

## 4. 最终判断

**当前维度适合作为“代码外观”量表**，核心假设成立：职责单一与否确实在屏幕上有肉眼可观察的外观表现（函数长度、异质块混杂、空行切割、命名体量比）。0–4 档形成单调递增的视觉谱系，但存在以下改进空间：

1. **5 档偏离了“职责边界”外观主线**，引入了模块内聚和函数列规律性，建议回退到更极致的职责外观描述（如“一次扫视可看完一个类中所有函数名即可理解全部意图”）。
2. **4 档与 5 档视觉差别不足**，“俳句集合”与“短小规律的函数列”对肉眼而言几乎是同一画面。
3. **3 档中“命名精准”带有语义判断**，建议替换为纯外观描述（如“每个块的名称短而具体，不需要注释即可辨识其意图”是可观察的命名外观，但“精准”本身需要阅读理解）。
